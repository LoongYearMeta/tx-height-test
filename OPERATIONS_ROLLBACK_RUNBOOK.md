# 版本发布、快照与回滚工程实施方案

## 1. 建设目标

建立一套独立于开发实现的生产操作体系：任何新版本不可用时，值班人员不改代码、不改 ledger、不现场分析即可在规定时间内切回最后可用版本，并撤销失败版本对受控文件、运行记录、队列和测试节点造成的改变。

目标指标：

- 回滚决策时间：业务硬指标异常后 5 分钟内。
- 停止失败版本写入：1 分钟内。
- 切回最后可用版本（LKG）：10 分钟内。
- 受控状态恢复点：发布切换前最后一个一致性快照，RPO 为 0。
- 每次生产发布前，回滚包和快照恢复必须已在预演环境验证。

## 2. 当前环境与基础设施选择

当前项目位于 ext4 文件系统，底层为 LVM：

```text
/dev/mapper/ubuntu--vg-ubuntu--lv  ext4
```

因此工程上采用：

- 不可变发布目录保存程序和依赖。
- systemd 管理蓝、绿两个运行槽位。
- 单独的 LVM 逻辑卷保存业务状态。
- LVM snapshot 提供切换前一致性快照。
- candidate 使用生产节点数据卷的只读克隆或独立测试节点，不能直接污染生产节点。

不应长期对当前根逻辑卷直接做生产回滚。实施时应先划分独立的应用状态卷和节点数据卷，使快照、恢复和容量管理相互隔离。

## 3. 生产目录和逻辑卷布局

目标目录：

```text
/srv/tx-height/
  releases/<release-id>/       # 不可变程序、依赖、配置模板
  slots/blue                   # 指向某个 release
  slots/green                  # 指向某个 release
  active                       # 当前生产槽位
  lkg                          # 最后可用 release
  manifests/                   # 发布和快照清单
  rollback/                    # 已验证的回滚包

/var/lib/tx-height/            # 独立 LVM：txheight-state
  ledger/
  output/
  runtime/

/var/lib/tbcnode/              # 独立 LVM：tbcnode-data
/var/log/tx-height/
/mnt/tx-height-snapshots/      # 快照只读挂载与验证位置
```

每个 release 必须包含锁定后的程序、`node_modules`、启动命令、非密钥配置快照、Git 提交号、Node.js 版本和文件校验清单。生产机禁止在回滚时执行 `npm install` 或从 Git 临时拼装版本。

## 4. 角色与权限

每次发布至少指定三种职责；人员不足时可以一人兼任，但发布批准人与执行人不能是同一个角色。

| 角色 | 职责 |
|---|---|
| 发布负责人 | 准备 release、发布清单、变更范围和回滚包 |
| 运维执行人 | 创建快照、切换槽位、执行健康检查和回滚 |
| 发布批准人 | 检查证据，批准上线、扩量、LKG 变更和快照删除 |

只有运维执行账户可以控制 systemd 服务、LVM 快照和生产 RPC 凭据。candidate 默认拿不到生产 RPC 凭据。

## 5. 一次性基础设施建设

按以下顺序建设，不涉及业务功能开发：

1. 统计 ledger、output 和节点数据目录的实际容量及每小时增长量。
2. 在 VG 中为 `txheight-state`、`tbcnode-data` 和快照预留空间；快照池不得与日常磁盘余量混用。
3. 安排停机窗口，把现有状态和节点数据迁移到独立逻辑卷。
4. 创建 `tx-height@blue.service` 和 `tx-height@green.service`，两个服务使用相同程序接口但不同运行目录、PID 文件和日志标识。
5. 创建独立 candidate 节点服务、RPC 端口和数据卷；网络策略禁止 candidate 凭据访问生产节点。
6. 建立发布清单、快照清单、切换记录和回滚记录的固定存放目录。
7. 建立监控面板：进程状态、真实 mempool 字节增长、成功交易数、错误率、任务积压、最近业务进度时间。
8. 完成一次“空版本”蓝绿切换和完整快照恢复演练，测量真实 RTO。

基础设施演练没有通过前，停止任何新版本生产发布。

## 6. 发布包制作

发布负责人在非生产工作目录制作 release：

1. 从明确 Git 提交构建完整目录，release-id 使用时间加提交短哈希。
2. 固化依赖，不在生产切换期间安装依赖。
3. 保存 Node.js 版本、启动参数和配置版本；密钥只记录引用，不复制明文。
4. 对整个 release 生成 SHA-256 校验清单。
5. 打包后设为只读，上传到 `/srv/tx-height/releases/<release-id>`。
6. 在预演机从这个确切目录启动，不允许从源码工作区启动。
7. 生成发布清单，明确：current、LKG、candidate、负责人、回滚触发阈值和回滚目标快照。

发布包一旦进入预演或生产不得原地修改；任何修改都必须产生新的 release-id。

## 7. 切换前一致性快照

运维执行人在维护窗口执行：

1. 暂停新任务入口。
2. 停止当前生产服务，并确认进程及其子进程全部退出。
3. 正常停止受控生产节点，使 mempool 和节点状态刷盘。
4. 执行 `sync`，记录 current release、LKG、区块高度、ledger 校验值、output 校验摘要和时间。
5. 分别为 `txheight-state` 和 `tbcnode-data` 创建带同一 snapshot-id 的 LVM snapshot。
6. 将两个 snapshot 只读挂载，检查文件系统并核对清单。
7. 写入 `/srv/tx-height/manifests/<snapshot-id>.json`，记录两个 LV snapshot 名称和全部边界信息。
8. 验证成功后恢复生产节点；未验证成功则取消发布并恢复原服务。

严禁只快照 ledger、不快照 output；也严禁把不同时间点的应用状态快照和节点快照组合成一个恢复点。

## 8. Candidate 预演与生产切换

### 隔离预演

1. 从切换前节点 snapshot 克隆 candidate 节点数据卷。
2. 从应用状态 snapshot 克隆 candidate 状态目录。
3. candidate 只连接克隆节点，执行完整工作负载。
4. 记录真实 mempool 增长、错误率、任务推进和资源占用。
5. 强制终止 candidate，再次启动，检查其恢复行为。
6. 废弃整个 candidate 克隆，确认生产状态没有变化。

### 生产切换

只有隔离预演通过后才允许：

1. 将 candidate release 配置到非 active 槽位。
2. 再次确认切换前 snapshot 为 `verified`。
3. 停止 current 槽位。
4. 原子切换 `active` 指针并启动 candidate 槽位。
5. 先开放预定的小流量窗口，不立即接管全部积压任务。
6. 观察真实业务指标；健康检查或进程存活不能替代业务指标。
7. 达到观察期且批准人签字后，candidate 才成为 current；原 current 保持为 LKG。

## 9. 自动停止与人工回滚阈值

每次发布清单必须填写具体阈值。以下任一条件发生时，监控先自动停止 candidate，值班人员直接执行回滚：

- 连续 5 分钟真实 mempool 字节无增长。
- 15 分钟平均吞吐低于 LKG 同等负载基线的 50%。
- 连续错误超过发布清单上限。
- 任务积压持续增长而完成数不增长。
- candidate 修改了不在 canary 清单内的任务。
- ledger、output 或节点边界校验异常。
- 运维执行人无法在 5 分钟内判断 candidate 是否继续安全运行。

触发后禁止在线改代码、手工改 ledger、清零重试或延长观察时间。

## 10. 标准快速回滚 Runbook

值班人员只按以下顺序执行：

1. 宣布回滚并记录开始时间、失败 release、LKG 和目标 snapshot-id。
2. 关闭任务入口，停止 blue/green 两个应用服务并确认无残留子进程。
3. 停止生产节点。
4. 对失败现场的应用状态卷和节点卷创建事故 snapshot，保留证据。
5. 卸载当前状态卷；将切换前 `verified` snapshot 恢复为新的可写卷，禁止覆盖原 snapshot。
6. 同样恢复与其 snapshot-id 相同的节点数据卷。
7. 核对 release、配置、ledger、output、节点高度及清单校验值全部属于同一恢复点。
8. 将 `active` 原子切换到 LKG release。
9. 先启动节点并完成 RPC/高度检查，再启动 LKG 应用服务。
10. 开放任务入口，观察真实 mempool 增长和任务完成情况。
11. 达到 LKG 基线后宣布回滚完成，记录实际 RTO。

任何校验不一致时停止恢复，不允许从其他快照手工补文件。

## 11. 外部影响能否撤销的硬边界

本地状态、输出文件和受控测试节点可以通过协调快照整体恢复。以下影响不能由本机快照撤销：

- 已广播到不受控制的远程节点的交易。
- 已传播到其他节点的 mempool 交易。
- 已经写入区块链的交易。

因此，candidate 在成为 current 前必须只使用隔离节点。如果业务要求 candidate 直接接触远程生产节点，就不能承诺“失败影响可撤销”，该发布必须被拒绝。这里没有运维技巧可以替代隔离边界。

## 12. 快照保留与容量管理

- LKG 切换前 snapshot：保留到 LKG 退役后至少一个发布周期。
- 当前版本切换前 snapshot：始终保留。
- 失败现场 snapshot：事故关闭并书面批准后才能删除。
- 每周至少验证一次 snapshot 可只读挂载和清单一致。
- 每月至少完成一次完整恢复演练。
- snapshot COW 使用率达到 70% 告警，达到 85% 时停止发布，防止 snapshot 失效。
- 删除 snapshot 必须由批准人确认，不得由自动清理任务单独决定。

## 13. 变更与演练记录

每次发布和回滚必须留下：

- release-id、Git 提交、依赖和配置版本。
- current、LKG 和 candidate 对应关系。
- snapshot-id、LV 名称、创建时间、校验结果和容量。
- 发布人、执行人、批准人及时间。
- 每个操作步骤的开始时间、结束时间和结果。
- 业务指标基线、发布后指标和回滚后指标。
- 实际 RTO/RPO 与偏差说明。

没有完整记录的版本不得被标记为 LKG。
