## 错误恢复框架

任何恢复工作必须依次经过以下阶段，禁止跳级：

### 1. 冻结

- 停止相关进程及自动重启机制。
- 保存代码版本、配置、ledger、日志、输出文件和节点状态快照。
- 对证据计算校验值；原件只读保存，后续分析使用副本。

### 2. 定界

- 明确事故开始提交、结束提交、运行时间、进程、节点和数据目录。
- 分开记录“代码变化”“本地记录变化”“节点外部状态变化”。
- Git 回退只处理代码，绝不推断它能回退 ledger 或节点状态。

### 3. 只读审计

- 第一轮工具只能读取并生成报告，不得修改业务数据。
- 所有推断必须标注证据来源和不确定性。
- 日志中的 `sent`、`done`、进程状态和 ledger 状态均不能单独证明节点已接受交易。

### 4. 恢复设计

- 恢复方案必须列出每一类记录的判定条件、允许动作、禁止动作和回滚方式。
- 对无法证明的状态使用 `unknown/quarantined`，不得擅自归类为待生成、待广播或已完成。
- 任何数据迁移先生成独立候选文件和差异报告，不覆盖原 ledger。

### 5. 审批

- 报告、恢复规则、候选变更和影响范围必须由用户明确批准。
- 未批准前不得运行会改变 ledger、输出集合、节点内存池或 Git 历史的操作。

### 6. 小范围演练

- 在隔离数据副本和测试节点上演练。
- 必须覆盖正常完成、中途退出、重复交易、策略拒绝、RPC 故障和重启恢复。
- 演练失败即停止，不得把失败对象自动重新入队。

### 7. 执行与核验

- 生产执行必须可中断、可审计，并逐项记录实际变更。
- 以节点外部状态和业务结果核验，不以进程退出码或日志文案代替。
- 执行后保留原始证据、候选文件、最终文件和完整差异。

## 强制开发与验收准则

1. 语法检查只证明文件可解析，不属于流程验收。
2. 单元测试只证明被测试函数，不属于端到端验收。
3. 未在真实兼容节点完成“生成、广播、入池核验、异常退出、恢复续跑”的代码，不得声明可运行。
4. 状态机变更必须先定义状态不变量、转换前置条件和崩溃一致性。
5. 恢复代码不得凭进程退出状态直接回退业务状态。
6. 兼容两种格式必须分别验证，并验证二者同时存在时不会饥饿、越序或互相污染。
7. HTTP 状态、JSON-RPC 错误和交易策略拒绝必须分层处理，禁止仅凭 HTTP 500 推断节点过载。
8. 禁止让单个广播失败关闭无关的生成、健康检查或其他任务。
9. 性能目标必须用节点实际接收的交易数和字节数验收；文件大小、发送调用数和日志计数不能替代。
10. 任何会改写 ledger 的恢复脚本默认必须是 dry-run，并输出逐条差异；实际写入需要单独明确授权。
11. 未建立可靠证据前，宁可停止并标记未知，也不得自动重播、重生成或清零重试。
12. 失败实现必须明确记录为失败，不得用后续补丁掩盖其端到端未通过的事实。

## 恢复工作的停止条件

出现以下任一情况必须立即停止：

- 原始证据将被覆盖或无法备份。
- 无法区分代码状态、本地记录状态与节点状态。
- 恢复动作依赖未经验证的推断。
- 测试结果与生产目标不一致。
- 用户撤销或未给出修改外部状态的授权。
